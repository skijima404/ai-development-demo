# 追跡可能性運用モデルメモ

- status: draft
- created_at: 2026-03-30
- updated_at: 2026-03-30
- scope: repository-local note

## 目的
このデモリポジトリで採用している追跡可能性の運用モデルを、後から参照できるように軽量に残す。

## 現在の基本モデル
現在の基本チェーンは次の通りです。

1. `プロダクトビジョン`
2. `価値の流れ`
3. `意図`
4. `基盤提案` と `機能提案`
5. `ユーザー操作仕様`
6. `実装仕様`

## このモデルが必要な理由
このリポジトリでは、次の問いに答えられる構造が必要です。
- このプロダクトは何を実演しようとしているのか
- どの価値の流れを改善しようとしているのか
- どの変化を意図しているのか
- どの基盤資産が必要なのか
- どのようにレビューされるべきか
- 実装はどこまでを触るべきか

## 現時点の正本の原則
現時点の正本は、次のような更新可能な資産に置きます。
- `docs/product/vision.md`
- `docs/traceability/value-streams/`
- `docs/traceability/intents/`
- 現行の基盤提案 / 機能提案
- 現行の UI 仕様 / 実装仕様

ADR は、以前の方向がなぜ変わったかを残すためのものです。
現時点の正本の置き場にはしません。

## 資産境界ルール
このリポジトリでは次の資産クラスを分けて扱います。
- プロダクト正本
- 要件・設計の正本
- 運用ガイダンス
- 履歴としての意思決定
- 将来の本番資産
- 将来の検証資産

## AI ネイティブ開発で重要な理由
AI は内容だけでなく、構造からも意図を推測します。
現時点の正本、補助メモ、実装資産が混ざると、後続の生成やレビューの安定性が落ちます。

## 保留中の問い
- `traceability` という分類で今後も十分か
- コミットやデプロイの引き継ぎ前に必須とする成果物は何か
- デプロイ準備の証跡をこのリポジトリでどう表現するか
