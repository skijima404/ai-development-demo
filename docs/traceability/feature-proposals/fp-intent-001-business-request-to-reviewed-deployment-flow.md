## 意図
業務要求が、要件具体化、追跡可能な仕様化、AI 実装、人間レビュー、デプロイ準備までつながるデモ品質の作業フローを提供する。

## 問題
ワークフロー提案がないままデモアプリの実装だけを進めると、業務ユーザーに見せたい全体の開発の流れが伝わらなくなる。

## 成功基準
1. 将来のデモアプリ作業を、エンドツーエンドの流れの中で説明できる。
2. 人間レビューが、名目上ではなく構造上の段階として残る。
3. コミットとデプロイ準備がデリバリーフローの一部として見える。
4. AI コーディング支援エージェントが、コミット後の GitOps 操作を補助できる。

## スコープ
- 対象範囲:
  - 作業フローの骨格定義
  - 要求 / 仕様 / 実装 / レビュー の関係づけ
  - コミット / デプロイ引き継ぎの見せ方
  - AI による GitOps tag 切り替え / ロールバック補助の見せ方
- 対象外:
  - デモアプリ自体の最終 UI
  - インフラ固有のデプロイ自動化
  - 詳細なセキュリティ / ポリシーレビュー手順

## 制約
- 技術面:
  - 要件・設計資産と本番コードは分離する
- 運用面:
  - 業務ユーザーに理解できる流れを保つ
- デモ設計:
  - 全自動の魔法のようには見せない
  - AI が GitOps 操作を行う場合も、何を変えたかを Git 履歴と運用資産で説明できること

## 変更契約
- 許可される変更:
  - 作業フロー各段階の具体化
  - レビュー / デプロイ引き継ぎの詳細化
- 禁止される変更:
  - 人間レビューの除去
  - 仕様と実装を追跡不能にまとめること
- 承認が必要:
  - 人間の確認点なしの完全自律デプロイを前提にする変更
- 検証:
  - 価値の流れ、UI 仕様、将来の実装資産と整合していること
- 巻き戻し:
  - 骨格定義に誤解がある場合は、修正版提案に差し替える
